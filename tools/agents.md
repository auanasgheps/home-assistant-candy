# Candy Simply-Fi XOR Algorithm (Python Port)

This documentation is intended for an LLM (Large Language Model) or a developer who needs to analyze, extend, or maintain the `simplyfi.py` Python script. The script is a Python utility to fetch configurations and states from Candy Simply-Fi devices.

## Architectural Overview

Candy Simply-Fi appliances expose a TCP port (typically port 80). To communicate, they require a specifically formatted raw TCP/HTTP request.
The response generated by the device is not a cleartext payload, but a hexadecimal string (e.g., `4a5b6c...`) that represents a JSON payload encrypted using a **Repeating-key XOR cipher**.

The encryption/decryption key:
- Is **unique** for each device.
- Typically has a fixed length of **16 bytes**.
- Is composed of printable ASCII characters.

## The Communication Algorithm

The data fetching mechanism (`get_candy_simplyfi_data`) uses raw sockets instead of high-level HTTP libraries (like `requests`). This ensures compatibility with the often unstable or non-standard embedded web servers on board Candy network cards.
1. Sends: `GET /http-{method}.json?encrypted=1 HTTP/1.1\nHost: {ip}\n\nConnection: close\r\n\r\n`
2. Reads the entire TCP response.
3. Isolates the last block after the HTTP header by searching for the last `\r` or `\n`.
4. Converts the cleaned hexadecimal string into raw bytes (`bytes.fromhex()`).

## The `getkey` Algorithm (Known-Plaintext Attack)

The core functionality is extracting the private key using the `find_xor_key` function. Since the returned payload is standard JSON, we know beforehand some strings that *must* be present, such as `"{\"WiFiStatus\":\"0\""`.

The algorithm performs a Known-Plaintext attack:
1. **Sliding Window:** Iterates the known string (`search_bytes`) byte by byte along the entire extracted ciphertext.
2. **Key Fragment Extraction:** Applies the XOR operation `partial_key[j] = data[pp + j] ^ search_bytes[j]`. If the result contains non-printable characters, the window is incorrect, and it moves to the next one.
3. **Propagation and Rotation:** Given the assumption that the key is 16 bytes and sequentially repeated throughout the text:
   - If the `partial_key` fragment is correctly aligned, it is extended by testing plausible key lengths `kl` up to 16 for these devices.
   - Generates different permutations by rotating the calculated fragment (to find the alignment at index 0 of the original encrypted payload).
4. **Holistic Validation:** Using the rotatively generated key block, the entire encrypted payload is decoded. If the whole resulting JSON output is entirely composed of valid alphanumeric characters (`is_printable`), the validation is considered successful.
5. **Final Extraction:** Any padding (`0xff`) is discarded from the candidate key, and the 16-character ASCII array representing the local device's key is returned.

### Technical Details and Handled Edge Cases

* **`is_printable` check:** Verifies the absence of esoteric characters outside the standard ASCII bounds `0x20 - 0x7E`, explicitly permitting only newlines, tabs, and carriage returns.
* **17-byte string limit handling:** The Python script natively handles 17-byte Known-Plaintexts (e.g., `{"WiFiStatus":"0"`) by safely truncating them to form 16-byte frames to test during extrapolation, matching the actual 16-byte device keys.
* **Ignored Byte Removal (`0xff`):** In the unknown key-extension loop, we use `0xff` as padding to verify the offset; it is subsequently entirely ignored in the string buffer before being returned.
